# Upgrading to EQL 2.3

`2.3.0` is a breaking release. **Customers re-encrypt their data as part of the upgrade.** The crypto-side counterpart (`@cipherstash/protect` / `protect-ffi` / proxy) emits a new payload shape for ste_vec elements; EQL 2.3 reads only the new shape.

The `eql_v2` schema name, top-level type names, operator names, and the root-level payload structure are unchanged â€” but the Blake3 (`b3`) family of functions and the `b3` sv-element field are removed, and customer-owned indexes referencing `eql_v2.blake3(col)` must be rebuilt against `eql_v2.hmac_256(col, '<selector>')`.

## TL;DR

1. **Recommended recipe shifts from operator-class indexes to functional indexes** ([U-001](#u-001-functional-indexes-as-the-canonical-recipe)). Existing opclass indexes keep working; you don't have to migrate. New schemas should use functional indexes.
2. **Equality and hashing now require `hm` (hmac) on the column** ([U-002](#u-002-equality-and-hashing-require-hmac)). Previously, equality could silently fall through to a `NULL` comparison or to Blake3 on synthetic fixtures. Now it raises with a clear message.
3. **Root-level `b3` (Blake3) is no longer consulted** ([U-003](#u-003-blake3-removed-at-root)). It was never emitted by `@cipherstash/protect` in production â€” only by test fixtures.
4. **ste_vec element equality term migrated from `b3` to `hm`; the entire `eql_v2.blake3` family is removed** ([U-004](#u-004-sv-element-equality-term-is-hm-not-b3)). A new `eql_v2.hmac_256(val, selector)` overload provides the canonical field-level equality extractor.
5. **Range operators (`<`, `<=`, `>`, `>=`) require `ob` (Block ORE) on the column** ([U-005](#u-005-range-operators-are-block-ore-only)). The operators are now inlinable SQL whose body extracts the `ob` term on both sides, so bare-form range queries engage a functional ORE index. Columns carrying only `oc` (sv-element CLLW ORE) terms must rewrite to the extractor form.
6. **ste_vec ORE-CLLW payload fields consolidate to a single `oc`** ([U-006](#u-006-stevec-ore-field-consolidation)). The pre-2.3 `ocf` / `ocv` split collapses to one `oc` field, with width carried on the ciphertext via a leading domain-tag byte. Function families `eql_v2.{has,compare,}_ore_cllw_{u64_8,var_8}` collapse to `eql_v2.{has,compare,}_ore_cllw`.

## Upgrade notes

### U-001: Functional indexes as the canonical recipe

**What changed.** The recommended index recipe has shifted from per-column operator-class indexes to per-column functional indexes. PR [#193](https://github.com/cipherstash/encrypt-query-language/pull/193) made `=`, `<>`, `~~` (`LIKE`), and `~~*` (`ILIKE`) inlinable so the planner can structurally match them against functional indexes â€” meaning bare-form queries (`WHERE col = $1`) now engage the index without any per-query rewriting:

```sql
-- Recommended in 2.3+
CREATE INDEX users_email_hmac_idx   ON users USING hash (eql_v2.hmac_256(email_encrypted));
CREATE INDEX users_email_bloom_idx  ON users USING gin  (eql_v2.bloom_filter(email_encrypted));
CREATE INDEX users_data_stevec_idx  ON users USING gin  (eql_v2.ste_vec(data_encrypted));
```

```sql
-- Still works in 2.3, discouraged for new schemas
CREATE INDEX users_email_idx ON users (email_encrypted eql_v2.encrypted_operator_class);
```

**Why.** Three reasons:

1. **Supabase compatibility.** `CREATE OPERATOR CLASS` requires superuser; functional indexes don't.
2. **No btree row-size ceiling.** Operator-class indexes index the full encrypted payload. On rich SEM term sets this hits Postgres's `index row size N exceeds btree version 4 maximum 2704` error. Functional indexes only store the extracted index term.
3. **Smaller, faster indexes.** Indexing `hmac_256(col)` instead of the full payload typically produces indexes an order of magnitude smaller.

**Action required.** None. Existing opclass indexes continue to work. If you want to capture the wins above for an existing column, the migration is non-destructive â€” both index types can coexist:

```sql
-- 1. Create the functional index alongside the opclass index
CREATE INDEX CONCURRENTLY users_email_hmac_idx
  ON users USING hash (eql_v2.hmac_256(email_encrypted));

-- 2. Verify the planner picks it
EXPLAIN ANALYZE SELECT * FROM users WHERE email_encrypted = $1;
-- Look for: Index Scan using users_email_hmac_idx

-- 3. Drop the opclass index when you're confident
DROP INDEX CONCURRENTLY users_email_idx;
```

**Out of scope.** `ORDER BY` over Block ORE columns still uses operator-class infrastructure â€” Postgres needs a custom comparator and there's no functional-index equivalent yet. Keep opclass indexes on columns you sort on.

### U-002: Equality and hashing require hmac

**What changed.** Two related operations are now strictly hmac-based at the root:

- **`a = b` and `a <> b` on `eql_v2_encrypted`** require both sides to carry `hm` (hmac_256). Without it, the comparison returns NULL and the query returns zero rows.
- **`eql_v2.hash_encrypted(value)`** (used by `GROUP BY`, `DISTINCT`, hash joins) requires the value to carry `hm`. The previous Blake3 fallback has been removed; the function raises with a clear message when `hm` is missing.

**Why.** Pre-2.3, equality could route through hmac *or* Blake3 depending on what was present in the payload, which made plans non-deterministic and made it impossible to reason about which index would be engaged. The Blake3 fallback only activated on synthetic test fixtures â€” `@cipherstash/protect` never emitted root-level `b3` in production. Removing the fallback aligns the documented contract with what production data has always looked like.

**Action required.** Audit any column you query with `=`, `<>`, `GROUP BY`, `DISTINCT`, or hash joins. Each must carry `hm` in its payload. If you're using `@cipherstash/protect`, configure the column with the `unique` index type â€” that's what emits `hm`. If `unique` is already present, you're done.

**Behavioural change to be aware of.** The error surface differs by operation:

- **`WHERE col = $1` / `WHERE col <> $1` on a column lacking `hm`**: silently returns zero rows. The `=` / `<>` operators inline to `hmac_256(a) = hmac_256(b)`; with `hm` missing, both extractors return NULL and the comparison evaluates to NULL â€” false in a WHERE context. Same shape as pre-2.3, but now deliberate and consistent rather than an accident of the dispatch chain.
- **`GROUP BY col` / `DISTINCT col` / hash joins**: raises loudly with a message naming the missing index term. The `hash_encrypted` function checks for `hm` explicitly and refuses to hash when it's absent.

The asymmetry is intentional: equality queries fail open (zero rows) because that's the only behaviour an inlinable `LANGUAGE sql` extractor can express; hash operations fail loud because they go through a plpgsql guard that can raise. Use a hash operation (`GROUP BY 1`, `SELECT count(DISTINCT col)`) in staging to surface misconfigured columns before promoting.

### U-003: Blake3 removed at root

**What changed.** Blake3 (`b3`) is no longer consulted at the payload root by `eql_v2.compare()` or by hash operations.

**Why.** Production `@cipherstash/protect` already operated this way â€” root-level `b3` was a fixture-only artefact.

**Action required.** None for production users. If you have:

- **Custom payloads or test fixtures that emit root-level `b3`** â€” drop that field. The bundled `create_encrypted_json` test helper has been updated. Move equality coverage to `hm` (U-002).
- **Application code that reads root-level `b3`** â€” none should exist; nothing in the documented protocol ever exposed it.

**Superseded.** The ste_vec-internal Blake3 path that this note previously described as "still in use" has now been removed entirely â€” see [U-004](#u-004-sv-element-equality-term-is-hm-not-b3) for the migration.

### U-004: sv element equality term is `hm`, not `b3`

**What changed.** ste_vec elements carry `hm` (HMAC-SHA256) as their selector-scoped equality term. The `b3` (Blake3) field is no longer emitted by the crypto layer and is no longer read by EQL. The entire `src/blake3/` module â€” `eql_v2.blake3()`, `eql_v2.has_blake3()`, `eql_v2.compare_blake3()`, and the `eql_v2.blake3` domain type â€” has been deleted from EQL.

A new function â€” `eql_v2.hmac_256(val eql_v2_encrypted, selector text)` â€” provides the canonical field-level equality extractor. It mirrors the root-level `eql_v2.hmac_256(val)` and composes with the same hash/index/equality recipes.

The `eql_v2.jsonb_path_query`, `jsonb_path_query_first`, and `jsonb_path_exists` functions are now inlinable SQL (previously plpgsql with `SET search_path`). Bodies are a single `SELECT` over `jsonb_array_elements`, so the planner can fold them into the calling query and remove per-row function call overhead on large ste_vec scans.

**Why.** After [U-002](#u-002-equality-and-hashing-require-hmac) tightened root-level equality to require `hm`, field-level equality through `->` / `jsonb_path_query_first` raised because sv elements carried `b3` only. Patching root-level `=` / `hash_encrypted` to fall back on `b3` measured 30â€“250Ă— planner regressions â€” it defeats functional-index match and merge-join sort-key hoist. The fix lives upstream: emit `hm` at the sv element level so the extracted value already carries the equality term, and drop the now-redundant `b3` path entirely.

**Action required.**

1. **Re-encrypt your data.** Use a `@cipherstash/protect` (and matching proxy) release that emits `hm` at sv element level. EQL 2.3 will not read old (b3-bearing) payloads correctly â€” `ste_vec_contains` returns false for the old shape, and equality on extracted values returns NULL.
2. **Drop any indexes built on `eql_v2.blake3(col)` before upgrading.** EQL 2.3 removes the `eql_v2.blake3` function; an existing index referencing it will fail at install time.

   ```sql
   -- Pre-upgrade, identify the indexes
   SELECT indexname, indexdef
   FROM pg_indexes
   WHERE indexdef LIKE '%eql_v2.blake3%';

   -- Drop each one
   DROP INDEX <indexname>;
   ```

3. **Rebuild the dropped indexes against `eql_v2.hmac_256(col, '<selector>')`** after re-encryption. The selector value for a given JSONB path is the same selector hash that `@cipherstash/protect` emits in the `s` field of the corresponding sv element. Two recipes exist for different use cases:

   ```sql
   -- (a) Hot single selector â€” fastest for a known equality predicate.
   --     One index per selector you care about.
   CREATE INDEX users_data_email_hmac_idx
     ON users USING hash (eql_v2.hmac_256(data_encrypted, '<selector-for-email>'));

   -- (b) Any selector â€” one GIN index covers all field-level equality
   --     queries on the column. Recommended for the proxy rewrite path
   --     where `col -> 'foo' = $1` predicates can target any selector.
   CREATE INDEX users_data_hmac_terms_idx
     ON users USING gin (eql_v2.hmac_256_terms(data_encrypted));

   -- Query shape against the GIN index:
   SELECT * FROM users
     WHERE eql_v2.hmac_256_terms(data_encrypted)
        @> '[{"s":"<selector-hash>","hm":"<hmac-hash>"}]'::jsonb;
   ```

4. **Update query recipes.** Where you used `eql_v2.jsonb_path_query_first(col, 'sel')` *only to extract a value for hash/equality*, switch to `eql_v2.hmac_256(col, 'sel')` for single-selector predicates or `eql_v2.hmac_256_terms(col) @> $1` for the proxy-rewrite path. `jsonb_path_query_first` remains the right tool when you actually want the encrypted value back (e.g. projection followed by decryption).

**Module organisation note.** The new functions `eql_v2.hmac_256(val, selector)` and `eql_v2.hmac_256_terms(val)` live in `src/jsonb/functions.sql` (the module that owns ste_vec-iterating extractors). The root-level `eql_v2.hmac_256(val)` and `eql_v2.hmac_256(val jsonb)` overloads remain in `src/hmac_256/functions.sql` and are now also inlinable SQL â€” see the U-002 amendment for the behaviour change.

**Behavioural change to be aware of.** Failure modes differ by operation when a column hasn't been re-encrypted (see [U-002](#u-002-equality-and-hashing-require-hmac) for the exact equality / hashing semantics):

- **`GROUP BY` / `DISTINCT` / hash joins** raise loudly via `eql_v2.hash_encrypted` with a message naming the missing `hm` term â€” the canonical smoke test for "is this column re-encrypted?".
- **`WHERE col = $1` / `WHERE col <> $1`** silently return zero rows because the inlined `=` extractor reduces to a NULL comparison.
- **`WHERE col @> $1`** on ste_vec containment silently produces empty matches when the element-level `hm` is absent â€” the comparator falls through `compare_hmac_256` to the literal-jsonb tiebreaker on missing terms.

Test the migration on a staging copy before promoting. Cover both axes: run a hash operation (`SELECT count(DISTINCT col) FROM t`) to surface misconfigured columns loudly, and run representative containment / equality queries to confirm they return the expected rows.

**Versioning note.** Per `CLAUDE.md`, payload-format changes typically warrant a major-version bump. The team has opted to land this as `2.3.0` because the public API surface (`eql_v2` schema name, top-level type names, operator names) is preserved â€” only the internal ste_vec payload shape and the Blake3 helpers change. Future readers comparing 2.3.0 against the project's versioning ladder should treat this as the documented exception.

### U-005: Range operators are Block ORE only

**What changed.** `<`, `<=`, `>`, `>=` on `eql_v2_encrypted` are now inlinable SQL functions whose bodies reduce to a direct comparison on the `ob` (Block ORE) term: `eql_v2.ore_block_u64_8_256(a) <op> eql_v2.ore_block_u64_8_256(b)`. The planner can structurally match `WHERE col < $1` against a functional btree index on `eql_v2.ore_block_u64_8_256(col)` (using the existing `eql_v2.ore_block_u64_8_256_operator_class`, which is `DEFAULT FOR TYPE`), so range queries engage that index without per-query rewriting.

The old plpgsql wrappers walked `eql_v2.compare()`'s priority list (Block ORE â†’ CLLW ORE â†’ hmac â†’ literal fallback). After this change, `<` / `<=` / `>` / `>=` no longer consult that priority list â€” they go straight to the `ob` extractor, which raises with `Expected an ore index (ob) value in json: â€¦` if the column doesn't carry one.

**Why.** Same precedent as U-002 for equality: make the canonical functional index match through bare-form predicates without the operator-class detour. Block ORE is the standard range encoding emitted by the crypto layer for numeric / timestamp / integer columns, so this matches the default configuration most callers run. The narrower contract is also what lets `ORDER BY col LIMIT n` get a fast Top-N sort: with the inlined operator, each comparison reduces to an inlined ORE-term comparison rather than a full plpgsql `compare()` dispatch.

**Action required.**

- **Columns configured with `ore` (default for numerics / timestamps / integers)** â€” no change. They carry `ob`, so the natural form is now both index-friendly and fast.
- **Columns configured with CLLW ORE (the consolidated `oc` field on sv elements)** â€” bare-form range queries on these columns will now raise. Two paths forward:
  - **Preferred:** migrate the column configuration to `ore` (Block ORE) so the natural form works everywhere.
  - **If you must keep the existing encoding:** rewrite range queries to the extractor form: `WHERE eql_v2.ore_cllw(e->'<selector>'::text) < eql_v2.ore_cllw($1::jsonb)`. The extractor is inlinable single-statement SQL, but functional-index match still requires an operator class on `eql_v2.ore_cllw` (tracked separately â€” see issue #220).
- **Selector-extracted comparisons** (`WHERE e->'selector' < $1`) â€” same rule applies recursively. If the extracted sub-payload doesn't carry `ob`, rewrite to the extractor matching the sub-payload's term type.

**Notes for ORDER BY.** Bare-form `ORDER BY col` still needs a Sort node â€” the natural-form sort key doesn't syntactically match a functional ORE index expression â€” but the residual Sort step is now fast because each comparison uses the inlined ORE-term path. To skip the Sort node entirely, write the ORDER BY in extractor form: `ORDER BY eql_v2.ore_block_u64_8_256(col)`. That sort key matches the functional index and lets PostgreSQL stream rows out of the index in order.

**Wider coverage is deferred.** A CASE-style operator body that re-introduces CLLW support under the same inlined operators is being considered for a future release. Until then, those term types route through the extractor form. OPE-based ordered comparison is moving to a separate encrypted column type â€” not in scope for `eql_v2_encrypted`.

### U-006: ste_vec ORE field consolidation

**What changed.** The pre-2.3 ORE-CLLW split into `ocf` (fixed-width, numeric) and `ocv` (variable-width, text) collapses to a single `oc` field on sv elements:

| Pre-2.3 field | Encoding | Post-2.3 home |
| --- | --- | --- |
| `ocf` | ORE CLLW, fixed-width (numeric) | `oc` |
| `ocv` | ORE CLLW, variable-width (text) | `oc` |

The numeric / text width distinction that the field name used to carry is now part of the ciphertext: each `oc` value is a hex string with a leading domain-tag byte (`0x00` numeric, `0x01` string) followed by the CLLW ciphertext. Walking the CLLW per-byte protocol over `[tag][ciphertext]` preserves cross-domain separation (numeric < string) and within-domain ordering, so a single column can hold a mix of numeric and string sv elements with a consistent total order.

The two ordered terms â€” `ob` (Block ORE, root scalars) and `oc` (CLLW ORE, sv elements) â€” are now mutually exclusive within a single payload or `sv` element.

**Why.** Matches the upstream consolidation in cipherstash-suite [#1955](https://github.com/cipherstash/cipherstash-suite/pull/1955) plus its follow-up ORE collapse. The runtime `ste_vec.mode` config knob (added in [#213](https://github.com/cipherstash/encrypt-query-language/pull/213)) accepts `standard` and `compat` for forward compatibility, but EQL 2.3's `eql_v2_encrypted` type handles only the ORE side (`standard` mode). OPE-based ordered comparison is moving to a separate encrypted column type in a future release; until then, configure ste_vec columns with `standard` mode.

**Strict type separation: `eql_v2_encrypted` is root-only; `eql_v2.ste_vec_entry` is the entry shape.** 2.3 introduces a new DOMAIN type `eql_v2.ste_vec_entry` over `jsonb`, constrained to the sv-element shape (an object that carries at minimum `s`). The previous misleading `(eql_v2_encrypted)` overloads of sv-element extractors (`eql_v2.ore_cllw`, `eql_v2.has_ore_cllw`, `eql_v2.compare_ore_cllw`) are removed; the replacements take `eql_v2.ste_vec_entry`. Pattern shift:

```sql
-- Pre-2.3 (the deprecated "fake-root" path):
WHERE eql_v2.ore_cllw(col -> 'sel') < eql_v2.ore_cllw($1::jsonb)

-- 2.3 (typed entry â€” explicit cast on the result of `->`):
WHERE eql_v2.ore_cllw((col -> 'sel').data::eql_v2.ste_vec_entry)
    < eql_v2.ore_cllw($1::jsonb)
```

The `->` operator still returns `eql_v2_encrypted` (so the merged-meta-plus-entry shape is unchanged for existing callers) â€” the explicit `.data::eql_v2.ste_vec_entry` cast reaches the new typed overload. The reason for the strict separation: per the v2.3 payload schema, `oc` is sv-element scope only and never appears at the root of a column value. The previous (eql_v2_encrypted) overloads worked only because `->` happened to merge sv-element fields into a fake-root shape; that wasn't load-bearing semantics, just an accident of the implementation.

Sv-element ordering also gets a typed 3-way comparator and a full operator suite on `eql_v2.ste_vec_entry Ă— eql_v2.ste_vec_entry`:

```sql
-- 3-way comparator (returns -1 / 0 / 1):
SELECT eql_v2.compare(
  (col -> 'sel').data::eql_v2.ste_vec_entry,
  ($1::jsonb)::eql_v2.ste_vec_entry
);

-- Boolean operators (=, <>, <, <=, >, >=) â€” inlinable, match functional
-- indexes built on `eql_v2.hmac_256(col -> 'sel')` or
-- `eql_v2.ore_cllw(col -> 'sel')` once the .data cast is applied:
WHERE (col -> 'sel').data::eql_v2.ste_vec_entry
    = ($1::jsonb)::eql_v2.ste_vec_entry;
```

**Action required.**

- **Re-encrypt your data** with a `@cipherstash/protect` (and matching proxy) release that emits `oc` instead of the legacy `ocf` / `ocv` split. Pre-2.3 payloads carrying `ocf` / `ocv` no longer satisfy `eql_v2.has_ore_cllw`.
- **Update any direct callers of the legacy function names.** The composite types `eql_v2.ore_cllw_u64_8` and `eql_v2.ore_cllw_var_8` are gone, replaced by `eql_v2.ore_cllw`. Function families collapse the same way: `eql_v2.has_ore_cllw_u64_8` â†’ `eql_v2.has_ore_cllw`, etc.
- **Update sv-element extractor calls** to use the typed overload via `.data::eql_v2.ste_vec_entry`. The previous `(eql_v2_encrypted)` overloads are removed (see pattern shift above).
- **Drop any custom indexes built on the legacy types** before upgrading. EQL 2.3 removes those types; an existing index referencing them will fail at install time. Re-create on the consolidated extractor.

## Verification checklist

- [ ] **`EXPLAIN ANALYZE` on representative queries** â€” equality, `LIKE`, jsonb path. Each plan should contain `Index Scan using <your_index>` rather than `Seq Scan`. Functional indexes engage automatically post-2.3 via the inlined operators.
- [ ] **No silent zero-row results.** Replay any query you suspect of being misconfigured. Post-2.3, missing `hm` raises rather than returning zero rows.
- [ ] **Smoke-test in staging.** The behavioural change in U-002 means previously-silent misconfigurations will now raise. Catch them before production.

## Rollback

`2.3.0` is binary-compatible with `2.2.x` data. Rolling back is just re-installing the prior release SQL. No data migration is required either way.

The application-level surface that's hardest to roll back is U-002's "raise instead of NULL" behaviour: code written against the 2.3 contract may not handle the silent-NULL semantics gracefully. If you've tightened error handling around equality, plan rollbacks accordingly.
