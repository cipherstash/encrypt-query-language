# Upgrading to EQL 2.3

`2.3.0` is a breaking release. **Customers re-encrypt their data as part of the upgrade.** The crypto-side counterpart (`@cipherstash/protect` / `protect-ffi` / proxy) emits a new payload shape for ste_vec elements; EQL 2.3 reads only the new shape.

The `eql_v2` schema name, top-level type names, operator names, and the root-level payload structure are unchanged — but the Blake3 (`b3`) family of functions and the `b3` sv-element field are removed, and customer-owned indexes referencing `eql_v2.blake3(col)` must be rebuilt against `eql_v2.eq_term(col -> '<selector>')`.

## TL;DR

1. **Recommended recipe shifts from operator-class indexes to functional indexes** ([U-001](#u-001-functional-indexes-as-the-canonical-recipe)). Existing opclass indexes keep working; you don't have to migrate. New schemas should use functional indexes.
2. **Equality and hashing now require `hm` (hmac) on the column** ([U-002](#u-002-equality-and-hashing-require-hmac)). Previously, equality could silently fall through to a `NULL` comparison or to Blake3 on synthetic fixtures. Now it raises with a clear message.
3. **Root-level `b3` (Blake3) is no longer consulted** ([U-003](#u-003-blake3-removed-at-root)). It was never emitted by `@cipherstash/protect` in production — only by test fixtures.
4. **ste_vec element equality term migrated from `b3` to `hm`; the entire `eql_v2.blake3` family is removed** ([U-004](#u-004-sv-element-equality-term-is-hm-not-b3)). The canonical field-level equality recipe is now the chained `eql_v2.eq_term(col -> '<selector>')`.
5. **Range operators (`<`, `<=`, `>`, `>=`) require `ob` (Block ORE) on the column** ([U-005](#u-005-range-operators-are-block-ore-only)). The operators are now inlinable SQL whose body extracts the `ob` term on both sides, so bare-form range queries engage a functional ORE index. Columns carrying only `oc` (sv-element CLLW ORE) terms must rewrite to the extractor form.
6. **ste_vec ORE-CLLW payload fields consolidate to a single `oc`** ([U-006](#u-006-ste_vec-ore-field-consolidation)). The pre-2.3 `ocf` / `ocv` split collapses to one `oc` field, with width carried on the ciphertext via a leading domain-tag byte. Function families `eql_v2.{has,compare,}_ore_cllw_{u64_8,var_8}` collapse to `eql_v2.{has,compare,}_ore_cllw`.
7. **`->` selector lookup returns a typed `eql_v2.ste_vec_entry`** ([U-007](#u-007-typed-arrow-selector)). The chained recipes `WHERE col -> 'sel' = $1::eql_v2.ste_vec_entry` (uses XOR-aware `eq_term`) and `ORDER BY eql_v2.ore_cllw(col -> 'sel')` engage functional indexes directly — no more synthetic-root cast workaround.
8. **`@>` containment accepts a typed `stevec_query` needle** ([U-008](#u-008-stevec-query-typed-containment)). Recommended recipe: `WHERE col @> '{"sv":[…]}'::eql_v2.stevec_query`. The DOMAIN explicitly forbids `c` (ciphertext) fields on the needle — a containment query matches indexes, not ciphertexts.

## Upgrade notes

### U-001: Functional indexes as the canonical recipe

**What changed.** The recommended index recipe has shifted from per-column operator-class indexes to per-column functional indexes. PR [#193](https://github.com/cipherstash/encrypt-query-language/pull/193) made `=`, `<>`, `~~` (`LIKE`), and `~~*` (`ILIKE`) inlinable so the planner can structurally match them against functional indexes — meaning bare-form queries (`WHERE col = $1`) now engage the index without any per-query rewriting:

```sql
-- Recommended in 2.3+
CREATE INDEX users_email_hmac_idx   ON users USING hash (eql_v2.hmac_256(email_encrypted));
CREATE INDEX users_email_bloom_idx  ON users USING gin  (eql_v2.bloom_filter(email_encrypted));
CREATE INDEX users_data_stevec_idx  ON users USING gin  (eql_v2.ste_vec(data_encrypted));
```

```sql
-- Still works in 2.3, discouraged for new schemas
CREATE INDEX users_email_idx ON users (email_encrypted eql_v2.encrypted_operator_class);
```

**Why.** Three reasons:

1. **Supabase compatibility.** `CREATE OPERATOR CLASS` requires superuser; functional indexes don't.
2. **No btree row-size ceiling.** Operator-class indexes index the full encrypted payload. On rich SEM term sets this hits Postgres's `index row size N exceeds btree version 4 maximum 2704` error. Functional indexes only store the extracted index term.
3. **Smaller, faster indexes.** Indexing `hmac_256(col)` instead of the full payload typically produces indexes an order of magnitude smaller.

**Action required.** None. Existing opclass indexes continue to work. If you want to capture the wins above for an existing column, the migration is non-destructive — both index types can coexist:

```sql
-- 1. Create the functional index alongside the opclass index
CREATE INDEX CONCURRENTLY users_email_hmac_idx
  ON users USING hash (eql_v2.hmac_256(email_encrypted));

-- 2. Verify the planner picks it
EXPLAIN ANALYZE SELECT * FROM users WHERE email_encrypted = $1;
-- Look for: Index Scan using users_email_hmac_idx

-- 3. Drop the opclass index when you're confident
DROP INDEX CONCURRENTLY users_email_idx;
```

**Out of scope.** `ORDER BY` over Block ORE columns still uses operator-class infrastructure — Postgres needs a custom comparator and there's no functional-index equivalent yet. Keep opclass indexes on columns you sort on.

### U-002: Equality and hashing require hmac

**What changed.** Two related operations are now strictly hmac-based at the root:

- **`a = b` and `a <> b` on `eql_v2_encrypted`** require both sides to carry `hm` (hmac_256). Without it, the comparison returns NULL and the query returns zero rows.
- **`eql_v2.hash_encrypted(value)`** (used by `GROUP BY`, `DISTINCT`, hash joins) requires the value to carry `hm`. The previous Blake3 fallback has been removed; the function raises with a clear message when `hm` is missing.

**Why.** Pre-2.3, equality could route through hmac *or* Blake3 depending on what was present in the payload, which made plans non-deterministic and made it impossible to reason about which index would be engaged. The Blake3 fallback only activated on synthetic test fixtures — `@cipherstash/protect` never emitted root-level `b3` in production. Removing the fallback aligns the documented contract with what production data has always looked like.

**Action required.** Audit any column you query with `=`, `<>`, `GROUP BY`, `DISTINCT`, or hash joins. Each must carry `hm` in its payload. If you're using `@cipherstash/protect`, configure the column with the `unique` index type — that's what emits `hm`. If `unique` is already present, you're done.

**Behavioural change to be aware of.** The error surface differs by operation:

- **`WHERE col = $1` / `WHERE col <> $1` on a column lacking `hm`**: silently returns zero rows. The `=` / `<>` operators inline to `hmac_256(a) = hmac_256(b)`; with `hm` missing, both extractors return NULL and the comparison evaluates to NULL — false in a WHERE context. Same shape as pre-2.3, but now deliberate and consistent rather than an accident of the dispatch chain.
- **`GROUP BY col` / `DISTINCT col` / hash joins**: raises loudly with a message naming the missing index term. The `hash_encrypted` function checks for `hm` explicitly and refuses to hash when it's absent.

The asymmetry is intentional: equality queries fail open (zero rows) because that's the only behaviour an inlinable `LANGUAGE sql` extractor can express; hash operations fail loud because they go through a plpgsql guard that can raise. Use a hash operation (`GROUP BY 1`, `SELECT count(DISTINCT col)`) in staging to surface misconfigured columns before promoting.

### U-003: Blake3 removed at root

**What changed.** Blake3 (`b3`) is no longer consulted at the payload root by `eql_v2.compare()` or by hash operations.

**Why.** Production `@cipherstash/protect` already operated this way — root-level `b3` was a fixture-only artefact.

**Action required.** None for production users. If you have:

- **Custom payloads or test fixtures that emit root-level `b3`** — drop that field. The bundled `create_encrypted_json` test helper has been updated. Move equality coverage to `hm` (U-002).
- **Application code that reads root-level `b3`** — none should exist; nothing in the documented protocol ever exposed it.

**Superseded.** The ste_vec-internal Blake3 path that this note previously described as "still in use" has now been removed entirely — see [U-004](#u-004-sv-element-equality-term-is-hm-not-b3) for the migration.

### U-004: sv element equality term is `hm`, not `b3`

**What changed.** ste_vec elements carry `hm` (HMAC-SHA256) as their selector-scoped equality term. The `b3` (Blake3) field is no longer emitted by the crypto layer and is no longer read by EQL. The entire `src/blake3/` module — `eql_v2.blake3()`, `eql_v2.has_blake3()`, `eql_v2.compare_blake3()`, and the `eql_v2.blake3` domain type — has been deleted from EQL.

The canonical field-level equality recipe is now the chained `eql_v2.eq_term(col -> '<selector>')`. `eq_term` is XOR-aware (coalesces `hm` and `oc` as bytea), so a single functional index covers both hm-bearing and oc-bearing selectors. The earlier pre-release `eql_v2.hmac_256(val eql_v2_encrypted, selector text)` was a fused selector-match + hm-extract that bypassed the typed model and silently NULLed on oc-bearing selectors — it has been removed in favour of the typed chain.

The `eql_v2.jsonb_path_query`, `jsonb_path_query_first`, and `jsonb_path_exists` functions are now inlinable SQL (previously plpgsql with `SET search_path`). Bodies are a single `SELECT` over `jsonb_array_elements`, so the planner can fold them into the calling query and remove per-row function call overhead on large ste_vec scans.

**Why.** After [U-002](#u-002-equality-and-hashing-require-hmac) tightened root-level equality to require `hm`, field-level equality through `->` / `jsonb_path_query_first` raised because sv elements carried `b3` only. Patching root-level `=` / `hash_encrypted` to fall back on `b3` measured 30–250× planner regressions — it defeats functional-index match and merge-join sort-key hoist. The fix lives upstream: emit `hm` at the sv element level so the extracted value already carries the equality term, and drop the now-redundant `b3` path entirely.

**Action required.**

1. **Re-encrypt your data.** Use a `@cipherstash/protect` (and matching proxy) release that emits `hm` at sv element level. EQL 2.3 will not read old (b3-bearing) payloads correctly — `ste_vec_contains` returns false for the old shape, and equality on extracted values returns NULL.
2. **Drop any indexes built on `eql_v2.blake3(col)` before upgrading.** EQL 2.3 removes the `eql_v2.blake3` function; an existing index referencing it will fail at install time.

   ```sql
   -- Pre-upgrade, identify the indexes
   SELECT indexname, indexdef
   FROM pg_indexes
   WHERE indexdef LIKE '%eql_v2.blake3%';

   -- Drop each one
   DROP INDEX <indexname>;
   ```

3. **Rebuild the dropped indexes against `eql_v2.eq_term(col -> '<selector>')`** after re-encryption. The selector value for a given JSONB path is the same selector hash that `@cipherstash/protect` emits in the `s` field of the corresponding sv element. Two recipes exist for different use cases:

   ```sql
   -- (a) Hot single selector — fastest for a known equality predicate.
   --     One index per selector you care about. The XOR-aware extractor
   --     covers both hm-bearing and oc-bearing selectors.
   CREATE INDEX users_data_email_eq_term_idx
     ON users USING hash (eql_v2.eq_term(data_encrypted -> '<selector-for-email>'));

   -- (b) Any selector — one GIN index covers all field-level equality
   --     queries on the column. Recommended for the proxy rewrite path
   --     where `col -> 'foo' = $1` predicates can target any selector.
   CREATE INDEX users_data_hmac_terms_idx
     ON users USING gin (eql_v2.hmac_256_terms(data_encrypted));

   -- Query shape against the GIN index:
   SELECT * FROM users
     WHERE eql_v2.hmac_256_terms(data_encrypted)
        @> '[{"s":"<selector-hash>","hm":"<hmac-hash>"}]'::jsonb;
   ```

4. **Update query recipes.** Where you used `eql_v2.jsonb_path_query_first(col, 'sel')` *only to extract a value for hash/equality*, switch to the typed chain — either `WHERE col -> 'sel' = $1::eql_v2.ste_vec_entry` (operator form; planner picks the inlined `eq_term` extractor) or directly `eql_v2.eq_term(col -> 'sel')` if you need the bytea term. `jsonb_path_query_first` remains the right tool when you actually want the encrypted value back (e.g. projection followed by decryption).

**Module organisation note.** The new function `eql_v2.eq_term(eql_v2.ste_vec_entry)` lives in `src/ste_vec/eq_term.sql`. The root-level `eql_v2.hmac_256(val)` and `eql_v2.hmac_256(val jsonb)` overloads remain in `src/hmac_256/functions.sql` and are now also inlinable SQL — see the U-002 amendment for the behaviour change.

**Behavioural change to be aware of.** Failure modes differ by operation when a column hasn't been re-encrypted (see [U-002](#u-002-equality-and-hashing-require-hmac) for the exact equality / hashing semantics):

- **`GROUP BY` / `DISTINCT` / hash joins** raise loudly via `eql_v2.hash_encrypted` with a message naming the missing `hm` term — the canonical smoke test for "is this column re-encrypted?".
- **`WHERE col = $1` / `WHERE col <> $1`** silently return zero rows because the inlined `=` extractor reduces to a NULL comparison.
- **`WHERE col @> $1`** on ste_vec containment silently produces empty matches when the element-level `hm` is absent — the comparator falls through `compare_hmac_256` to the literal-jsonb tiebreaker on missing terms.

Test the migration on a staging copy before promoting. Cover both axes: run a hash operation (`SELECT count(DISTINCT col) FROM t`) to surface misconfigured columns loudly, and run representative containment / equality queries to confirm they return the expected rows.

**Versioning note.** Per `CLAUDE.md`, payload-format changes typically warrant a major-version bump. The team has opted to land this as `2.3.0` because the public API surface (`eql_v2` schema name, top-level type names, operator names) is preserved — only the internal ste_vec payload shape and the Blake3 helpers change. Future readers comparing 2.3.0 against the project's versioning ladder should treat this as the documented exception.

### U-005: Range operators are Block ORE only

**What changed.** `<`, `<=`, `>`, `>=` on `eql_v2_encrypted` are now inlinable SQL functions whose bodies reduce to a direct comparison on the `ob` (Block ORE) term: `eql_v2.ore_block_u64_8_256(a) <op> eql_v2.ore_block_u64_8_256(b)`. The planner can structurally match `WHERE col < $1` against a functional btree index on `eql_v2.ore_block_u64_8_256(col)` (using the existing `eql_v2.ore_block_u64_8_256_operator_class`, which is `DEFAULT FOR TYPE`), so range queries engage that index without per-query rewriting.

The old plpgsql wrappers walked `eql_v2.compare()`'s priority list (Block ORE → CLLW ORE → hmac → literal fallback). After this change, `<` / `<=` / `>` / `>=` no longer consult that priority list — they go straight to the `ob` extractor, which raises with `Expected an ore index (ob) value in json: …` if the column doesn't carry one.

**Why.** Same precedent as U-002 for equality: make the canonical functional index match through bare-form predicates without the operator-class detour. Block ORE is the standard range encoding emitted by the crypto layer for numeric / timestamp / integer columns, so this matches the default configuration most callers run. The narrower contract is also what lets `ORDER BY col LIMIT n` get a fast Top-N sort: with the inlined operator, each comparison reduces to an inlined ORE-term comparison rather than a full plpgsql `compare()` dispatch.

**Action required.**

- **Columns configured with `ore` (default for numerics / timestamps / integers)** — no change. They carry `ob`, so the natural form is now both index-friendly and fast.
- **Columns configured with CLLW ORE (the consolidated `oc` field on sv elements)** — bare-form range queries on these columns will now raise. Two paths forward:
  - **Preferred:** migrate the column configuration to `ore` (Block ORE) so the natural form works everywhere.
  - **If you must keep the existing encoding:** use the typed sv-entry chain. `e -> '<sel>'` returns `eql_v2.ste_vec_entry` post-U-007, so `WHERE e -> '<sel>'::text < $1::eql_v2.ste_vec_entry` inlines to `eql_v2.ore_cllw(a) < eql_v2.ore_cllw(b)` and `ORDER BY eql_v2.ore_cllw(e -> '<sel>'::text) LIMIT n` engages a functional btree on the same expression. The `eql_v2.ore_cllw_ops` btree opclass (DEFAULT FOR TYPE) wires this without an explicit opclass annotation on the index.
- **Selector-extracted comparisons** (`WHERE e->'selector' < $1`) — same rule applies recursively. If the extracted sub-payload doesn't carry `ob`, rewrite to the extractor matching the sub-payload's term type.

**Notes for ORDER BY.** Bare-form `ORDER BY col` still needs a Sort node — the natural-form sort key doesn't syntactically match a functional ORE index expression — but the residual Sort step is now fast because each comparison uses the inlined ORE-term path. To skip the Sort node entirely, write the ORDER BY in extractor form: `ORDER BY eql_v2.ore_block_u64_8_256(col)`. That sort key matches the functional index and lets PostgreSQL stream rows out of the index in order.

**Wider coverage is deferred.** A CASE-style operator body that re-introduces CLLW support under the same inlined operators is being considered for a future release. Until then, those term types route through the extractor form. OPE-based ordered comparison is moving to a separate encrypted column type — not in scope for `eql_v2_encrypted`.

### U-006: ste_vec ORE field consolidation

**What changed.** The pre-2.3 ORE-CLLW split into `ocf` (fixed-width, numeric) and `ocv` (variable-width, text) collapses to a single `oc` field on sv elements:

| Pre-2.3 field | Encoding | Post-2.3 home |
| --- | --- | --- |
| `ocf` | ORE CLLW, fixed-width (numeric) | `oc` |
| `ocv` | ORE CLLW, variable-width (text) | `oc` |

The numeric / text width distinction that the field name used to carry now lives in the plaintext: a domain-tag byte (`0x00` numeric, `0x01` string) is prepended to the plaintext input before CLLW encryption. The resulting `oc` is the encrypted form, but because CLLW ORE is order-preserving per-byte, comparing two `oc` values via the CLLW protocol preserves the relative ordering induced by that leading tag — numeric sorts before string, and within each domain the regular ordering applies. A single column can therefore hold a mix of numeric and string sv elements with a consistent total order.

The two ordered terms are **scope-disjoint**: `ob` (Block ORE) appears only on root scalars and is never present on sv elements; `oc` (CLLW ORE) appears only on sv elements and is never present at the root of a payload.

**Why.** Matches the upstream consolidation in cipherstash-suite (PR #1955 plus its follow-up ORE collapse). The runtime `ste_vec.mode` config knob (added in [#213](https://github.com/cipherstash/encrypt-query-language/pull/213)) accepts `standard` and `compat` for forward compatibility, but EQL 2.3's `eql_v2_encrypted` type handles only the ORE side (`standard` mode). OPE-based ordered comparison is moving to a separate encrypted column type in a future release; until then, configure ste_vec columns with `standard` mode.

**Strict type separation: `eql_v2_encrypted` is root-only; `eql_v2.ste_vec_entry` is the entry shape.** 2.3 introduces a new DOMAIN type `eql_v2.ste_vec_entry` over `jsonb`, constrained to the sv-element shape (an object carrying `s`, `c`, and exactly one of `hm`/`oc`). The previous misleading `(eql_v2_encrypted)` overloads of sv-element extractors (`eql_v2.ore_cllw`, `eql_v2.has_ore_cllw`, `eql_v2.compare_ore_cllw`) are removed; the replacements take `eql_v2.ste_vec_entry`.

In an earlier pre-release of 2.3, callers had to apply a `.data::eql_v2.ste_vec_entry` cast on the result of `->` to reach the typed overload. That cast is no longer needed — see [U-007](#u-007-typed-arrow-selector). The recipe is now:

```sql
-- 2.3 (typed entry — `->` returns ste_vec_entry directly):
WHERE eql_v2.ore_cllw(col -> 'sel') < eql_v2.ore_cllw($1::jsonb)
-- or, equivalently, via the entry-typed `<` operator:
WHERE col -> 'sel' < $1::eql_v2.ste_vec_entry
```

Sv-element comparison gets the full typed operator suite (`=`, `<>`, `<`, `<=`, `>`, `>=`) on `eql_v2.ste_vec_entry × eql_v2.ste_vec_entry`. Equality reduces to `eq_term(a) = eq_term(b)` (XOR-aware — see [U-007](#u-007-typed-arrow-selector)); ordering reduces to `ore_cllw(a) <op> ore_cllw(b)`.

**Action required.**

- **Re-encrypt your data** with a `@cipherstash/protect` (and matching proxy) release that emits `oc` instead of the legacy `ocf` / `ocv` split. Pre-2.3 payloads carrying `ocf` / `ocv` no longer satisfy `eql_v2.has_ore_cllw`.
- **Update any direct callers of the legacy function names.** The composite types `eql_v2.ore_cllw_u64_8` and `eql_v2.ore_cllw_var_8` are gone, replaced by `eql_v2.ore_cllw`. Function families collapse the same way: `eql_v2.has_ore_cllw_u64_8` → `eql_v2.has_ore_cllw`, etc.
- **Update sv-element extractor calls** to feed `eql_v2.ore_cllw` directly with `col -> 'sel'` (the typed return makes this work without a `.data` cast — see U-007). The previous `(eql_v2_encrypted)` overloads are removed.
- **Drop any custom indexes built on the legacy types** before upgrading. EQL 2.3 removes those types; an existing index referencing them will fail at install time. Re-create on the consolidated extractor.

### U-007: Typed `->` selector lookup

**What changed.** `->` on `eql_v2_encrypted` (all three overloads — text selector, encrypted selector, integer index) now returns `eql_v2.ste_vec_entry` (a DOMAIN over `jsonb` constrained to the sv-element shape). The previous return type — `eql_v2_encrypted` with a synthetic root merging `meta` + entry fields — is gone. The bodies are inlinable `LANGUAGE sql` (using `jsonb_path_query_first` for the text case and direct jsonb indexing for the integer case), so the planner can fold `col -> '<sel>'` into the calling query expression for functional-index matching.

A new XOR-aware extractor `eql_v2.eq_term(eql_v2.ste_vec_entry) RETURNS bytea` is the canonical equality term: it coalesces `hm` and `oc` (both deterministic), so one functional hash index on `eql_v2.eq_term(col -> '<sel>')` covers both hm-bearing selectors (bool leaves, array / object roots) and oc-bearing selectors (string / number leaves). The previous `eql_v2.hmac_256(col, '<sel>')` recipe — introduced as a pre-release migration path off Blake3 — is removed; the chained `eq_term` form is its successor.

The `=`/`<>` operators on `eql_v2.ste_vec_entry × eql_v2.ste_vec_entry` reduce to `eq_term(a) = eq_term(b)` (and `<>` likewise), so the bare-form predicate `WHERE col -> '<sel>' = $1::eql_v2.ste_vec_entry` engages a functional hash index built on `eql_v2.eq_term(col -> '<sel>')` without any per-query rewriting.

**Why.** The synthetic-root return was a long-standing design gap: an `eql_v2.ore_cllw` extractor read `oc` from a payload that — at the root — never has it; the operator chain worked only because `->` merged sv-element fields into a fake-root shape. Once `->` is typed at the sv-entry layer, the operator chain is direct, the functional-index match is structural, and the XOR contract on sv-element terms becomes expressible in the type system rather than scattered through dispatch code.

The earlier `hmac_256(col, '<sel>')` recipe was silently wrong on oc-bearing selectors (string / number leaves) — both sides extracted NULL and the predicate returned zero rows. `eq_term` fixes this by coalescing `hm` and `oc` as bytea; the two byte distributions are disjoint by construction (different keys, different protocols), so byte equality on the coalesce is unambiguous for any given selector.

**Action required.**

1. **Rewrite WHERE / GROUP BY predicates** that previously used the fused `hmac_256(col, '<sel>')`:

   | Pre-release | 2.3 |
   | --- | --- |
   | `eql_v2.hmac_256(col, '<sel>')` | `eql_v2.eq_term(col -> '<sel>'::text)` |
   | `WHERE eql_v2.hmac_256(col, '<sel>') = eql_v2.hmac_256(... 'sel')` | `WHERE col -> '<sel>'::text = $1::eql_v2.ste_vec_entry` |
   | `CREATE INDEX … USING hash (eql_v2.hmac_256(col, '<sel>'))` | `CREATE INDEX … USING hash (eql_v2.eq_term(col -> '<sel>'::text))` |

2. **Drop direct dependents of the old return type.** If application code consumed `col -> '<sel>'` as `eql_v2_encrypted` and read `.data` or passed it to root-level operators, it now sees `eql_v2.ste_vec_entry` instead. The typed value has no `.data` accessor (it's a domain over jsonb, not a composite). JSONB field access via `->>`, `->`, `?` etc. still works directly on the value.

3. **Order-by / range queries on selected entries** simplify to `ORDER BY eql_v2.ore_cllw(col -> '<sel>')` — no more `.data::eql_v2.ste_vec_entry` cast. The `eql_v2.ore_cllw_ops` btree opclass (DEFAULT FOR TYPE) wires the functional-index match without an explicit opclass annotation.

**Behavioural change to be aware of.** Pre-this-change, `WHERE col -> '<sel>' = $1` resolved to the root-level `eql_v2_encrypted` `=` operator on the synthetic-root return, which incidentally worked for hm-bearing selectors and silently returned zero rows for oc-bearing ones. Post-flip, the predicate types as `ste_vec_entry × ste_vec_entry` (or `eql_v2_encrypted × …` depending on RHS cast), so equality now correctly fires through `eq_term` for both selector kinds. If you were observing zero-rows on oc-bearing selectors and worked around it, you can drop the workaround.

### U-008: Typed `stevec_query` containment needle

**What changed.** 2.3 introduces `eql_v2.stevec_query` — a DOMAIN over `jsonb` constrained to the query-shaped sv payload (a `{"sv": [...]}` object whose elements carry selector + index terms but **never** a `c` (ciphertext) field). Two new operator overloads target this type for type-safe containment:

```sql
-- @>(eql_v2_encrypted, eql_v2.stevec_query) — recommended needle form
WHERE encrypted_doc @> '{"sv":[{"s":"<sel>","hm":"<hm>"}]}'::eql_v2.stevec_query;

-- @>(eql_v2_encrypted, eql_v2.ste_vec_entry) — convenience for single entries
WHERE encrypted_doc @> (other_doc -> '<sel>');

-- <@ mirror operators for both
```

A companion cast `eql_v2.to_stevec_query(eql_v2_encrypted)` strips `c` from every sv element so callers can convert another encrypted column into a containment needle:

```sql
SELECT a.*
  FROM docs a, docs b
 WHERE a.encrypted_doc @> b.encrypted_doc::eql_v2.stevec_query
   AND b.id = 42;
```

The existing `@>(eql_v2_encrypted, eql_v2_encrypted)` overload is preserved for back-compat with current call sites. The typed form is the recommended path going forward — the DOMAIN's CHECK constraint documents (and enforces) that a containment needle never carries a ciphertext.

**Why.** Containment matches against indexes (`s` + `hm`/`oc`), never against ciphertexts (`c`). The pre-release `@>(eql_v2_encrypted, eql_v2_encrypted)` shape was misleading at the API surface — it accepted full encrypted payloads as both container and needle, even though only the container's `c` is meaningful for the operation. Typing the needle as `stevec_query` makes the contract explicit: "this side is a query, not a value."

The DOMAIN CHECK also catches a common authoring mistake — assembling a needle by hand-copying an sv element and forgetting to strip `c`. Pre-flip, that worked because `ste_vec_contains` ignored `c` in the matching logic; post-flip, the cast raises with a clear CHECK violation.

**Action required.**

- **New code.** Use the typed needle for any new containment recipe.
- **Existing callers.** No change required — the untyped `@>(eql_v2_encrypted, eql_v2_encrypted)` overload still works. Consider migrating to the typed form when convenient; it's a one-line cast change at the call site.

## Verification checklist

- [ ] **`EXPLAIN ANALYZE` on representative queries** — equality, `LIKE`, jsonb path. Each plan should contain `Index Scan using <your_index>` rather than `Seq Scan`. Functional indexes engage automatically post-2.3 via the inlined operators.
- [ ] **No silent zero-row results.** Replay any query you suspect of being misconfigured. Post-2.3, missing `hm` raises rather than returning zero rows.
- [ ] **Smoke-test in staging.** The behavioural change in U-002 means previously-silent misconfigurations will now raise. Catch them before production.

## Rollback

`2.3.0` is binary-compatible with `2.2.x` data. Rolling back is just re-installing the prior release SQL. No data migration is required either way.

The application-level surface that's hardest to roll back is U-002's "raise instead of NULL" behaviour: code written against the 2.3 contract may not handle the silent-NULL semantics gracefully. If you've tightened error handling around equality, plan rollbacks accordingly.
