# Upgrading to EQL 2.4

`2.4.0` introduces the `eql_v2_int4` domain family — five jsonb-backed capability-encoded domains for encrypted `int4` columns. Additive change; no rename, no breaking removal.

## TL;DR

1. Pick a variant for each encrypted int4 column based on the operators you need and the index terms your Proxy emits:
   - `eql_v2_int4_ct` — storage only
   - `eql_v2_int4_eq` — HMAC equality (`=`, `<>`)
   - `eql_v2_int4_ord_ore` — HMAC + ORE-block ordering (range engages a btree operator class)
   - `eql_v2_int4_ord_ope` — HMAC + OPE-direct ordering (range engages a functional btree)
   - `eql_v2_int4` — default, identical to `_ord_ore`
2. Declare columns with the chosen variant type.
3. Attach the matching functional indexes (see the per-variant recipe in [`docs/reference/encrypted-int4-walkthrough.md`](../reference/encrypted-int4-walkthrough.md)).
4. Install EQL 2.4.

## Compatibility

| Component                                | Status                                                |
|------------------------------------------|-------------------------------------------------------|
| `eql_v2` schema name                     | **Unchanged.**                                        |
| `eql_v2_encrypted` composite type        | Unchanged.                                            |
| `eql_v2_int4*` domain family             | **New.** Five domains in `public`.                    |
| Function/operator names                  | New names follow the convention `eql_v2_int4_<variant>_<op>`. |
| Postgres 14, 15, 16, 17                  | Supported.                                            |

## Upgrade notes

### U-001: `eql_v2_int4` variant family

**What's new.** Five new domains in `public`:

- `eql_v2_int4_ct` (storage only — all bool operators raise)
- `eql_v2_int4_eq` (HMAC equality only — `=`, `<>`)
- `eql_v2_int4_ord_ore` (HMAC equality + ORE-block ordering — adds `<`, `<=`, `>`, `>=`)
- `eql_v2_int4_ord_ope` (HMAC equality + OPE-direct ordering — same operators as `_ord_ore` with functional-index-engaged range)
- `eql_v2_int4` (default, behaves as `_ord_ore`)

**Why a family rather than a single domain.** The variant name declares which payload terms the column must carry, which determines the operator surface and indexability. The old name `encrypted_int4` would have hidden these distinctions: a `_ord_ope` column must carry `opf`, a `_ord_ore` column must carry `ob`, a `_eq` column needs only `hm`. Putting the capability in the type name makes the contract checkable at the schema level.

The variant family enforces *operator policy* at the type level — every variant declares operator bindings for every operator. Unsupported operators dispatch to a variant-specific PL/pgSQL blocker that raises `operator <op> is not supported for <variant>`. The error surfaces once per query (for planner-time-foldable constant predicates) or once per row scanned (general case), never silently falling through to native jsonb comparison.

The variant family does **not** enforce payload-term presence at the type level. The domains are jsonb-backed with no CHECK constraints, so a column declared `eql_v2_int4_ord_ope` can hold a jsonb value missing `opf`. The mismatch surfaces at *query execution* when the extractor function (`eql_v2.ope_cllw_u64_65(jsonb)`, `eql_v2.hmac_256(jsonb)`, `eql_v2.ore_block_u64_8_256(jsonb)`) raises `RAISE EXCEPTION` per-row (e.g. `'Expected a ope_cllw_u64_65 index (opf) value in json'`). Schema authors must keep the variant name aligned with what their Proxy actually emits for that column; the type system does not catch drift.

**Range indexing (`_ord_ore`).** `_ord_ore` and the default `eql_v2_int4` use `eql_v2.compare_ore_block_u64_8_256`, which is PL/pgSQL. The range operator wrappers are therefore `LANGUAGE plpgsql IMMUTABLE` — deliberately non-inlinable, so the `<` / `<=` / `>` / `>=` operator nodes survive in the parse tree for the planner to match against a **btree operator class**. This is the same mechanism the core `eql_v2_encrypted` type uses for ORE (`src/operators/operator_class.sql`); a *functional* index on the extractor never engages, because the wrapper can't inline down to a planner-visible extractor expression.

Each variant ships an operator class — `eql_v2.eql_v2_int4_ord_ore_operator_class` and `eql_v2.eql_v2_int4_operator_class`. It **must be named explicitly** in `CREATE INDEX`:

```sql
CREATE INDEX orders_amount_ore_idx ON orders
USING btree (amount eql_v2.eql_v2_int4_ord_ore_operator_class);
```

A bare `USING btree (amount)` does **not** work: the column is a domain over `jsonb`, and PostgreSQL resolves a domain column's default operator class via its base type (`jsonb_ops`), never a domain-specific class. Naming the class is required, by design.

With the named index, range queries engage it (verified — `Bitmap Index Scan` / `Index Scan`).

**Supabase caveat.** Operator classes are unsupported on Supabase; the EQL Supabase build variant strips every `*operator_class.sql` file (`tasks/build.sh`). On Supabase, `_ord_ore` / default range queries fall back to seq-scan — there is no operator class to name. If you need indexed range on Supabase, use `eql_v2_int4_ord_ope`: its range wrappers reduce to inlineable bytea lex-compare and engage a *functional* btree (not stripped from the Supabase build).

**Domain ordering footgun.** `ORDER BY col` on a jsonb-backed domain follows native jsonb comparison, not the operator class — even when an operator-class index exists, because `ORDER BY col` requests the column type's *default* sort order, which (per the domain→base-type rule above) is `jsonb_ops`. Sort by the extractor expression instead:

```sql
-- _ord_ore / default: sort by ORE bytes
ORDER BY eql_v2.ore_block_u64_8_256(col::jsonb)
-- _ord_ope: sort by OPE bytes
ORDER BY eql_v2.eql_v2_int4_ord_ope_ope_key(col::jsonb)
```

## Verification checklist

- [ ] `\dT public.eql_v2_int4*` shows the five new domains.
- [ ] Sample equality query on each variant shows `Index Scan` on the hmac functional btree.
- [ ] Sample range query on `_ord_ope` shows `Index Scan` on the OPE-key functional btree.
- [ ] Sample range query on `_ord_ore`, indexed with `USING btree (col eql_v2.eql_v2_int4_ord_ore_operator_class)`, shows `Index Scan` / `Bitmap Index Scan` (`Seq Scan` on Supabase builds, which exclude operator classes).
