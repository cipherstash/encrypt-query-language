# Upgrading to EQL 2.4

`2.4.0` introduces one breaking change: the single `public.encrypted_int4` domain is replaced by a family of five capability-encoded variants.

## TL;DR

1. Identify every table column of type `public.encrypted_int4`.
2. Pick the variant whose operator surface matches what your Proxy emits:
   - `eql_v2_int4_ct` — storage only
   - `eql_v2_int4_eq` — HMAC equality (`=`, `<>`)
   - `eql_v2_int4_ord_ore` — HMAC + ORE-block ordering (range seq-scan)
   - `eql_v2_int4_ord_ope` — HMAC + OPE-direct ordering (range engages btree)
   - `eql_v2_int4` — default, identical to `_ord_ore`
3. Run `ALTER TABLE t ALTER COLUMN c TYPE <variant> USING c::jsonb::<variant>` for each.
4. Install EQL 2.4. The migration's `DROP DOMAIN public.encrypted_int4` now succeeds.

## Compatibility

| Component                         | Status                                                |
|-----------------------------------|-------------------------------------------------------|
| `eql_v2` schema name              | **Unchanged.** No rename.                             |
| `public.encrypted_int4` domain    | **Removed.** No compatibility alias.                  |
| `public.encrypted_text` domain    | Unchanged.                                            |
| `public.encrypted_jsonb` domain   | Unchanged.                                            |
| `eql_v2_encrypted` composite type | Unchanged.                                            |
| Function/operator names           | New names follow the convention `eql_v2_int4_<variant>_<op>`. |
| Postgres 14, 15, 16, 17           | Supported.                                            |

## Upgrade notes

### U-001: `encrypted_int4` rename

**What changed.** `public.encrypted_int4` is replaced by five new domains in `public`:

- `eql_v2_int4_ct` (storage only — all bool operators raise)
- `eql_v2_int4_eq` (HMAC equality only — `=`, `<>`)
- `eql_v2_int4_ord_ore` (HMAC equality + ORE-block ordering — adds `<`, `<=`, `>`, `>=`)
- `eql_v2_int4_ord_ope` (HMAC equality + OPE-direct ordering — same operators as `_ord_ore` with functional-index-engaged range)
- `eql_v2_int4` (default, behaves as `_ord_ore`)

`tasks/uninstall.sql` only drops the `eql_v2` schema, so `public.encrypted_int4` would normally survive uninstall — the 2.4 install migration explicitly `DROP DOMAIN IF EXISTS public.encrypted_int4`. The DROP fails loudly if any user table still depends on the old type; that's the intended migration signal.

**Why.** The old name hid which index terms a column carried. The variant name now declares it: a `_ord_ope` column must carry `opf`, a `_ord_ore` column must carry `ob`, a `_eq` column needs only `hm`.

The variant family enforces *operator policy* at the type level — every variant declares operator bindings for every operator (supported or not). Unsupported operators dispatch to a blocker function that raises `'operator <op> is not supported for <variant>'`. For example, `<` on `eql_v2_int4_eq` resolves to `eql_v2.eql_v2_int4_eq_lt(eql_v2_int4_eq, eql_v2_int4_eq)`, which is a PL/pgSQL blocker. The error surfaces once per query (PL/pgSQL planner-time-folded constant predicate) or once per row scanned (general case), never silently falling through to native jsonb comparison. The cost of declaring blocker bindings is paid in extra `CREATE FUNCTION`/`CREATE OPERATOR` statements at install; the benefit is a clear, variant-specific error rather than `operator does not exist`.

The variant family does **not** enforce payload-term presence at the type level. The domains are jsonb-backed with no CHECK constraints, so a column declared `eql_v2_int4_ord_ope` can hold a jsonb value missing `opf`. The mismatch surfaces at *query execution* when the extractor function (`eql_v2.ope_cllw_u64_65(jsonb)`, `eql_v2.hmac_256(jsonb)`, `eql_v2.ore_block_u64_8_256(jsonb)`) raises `RAISE EXCEPTION` per-row (e.g. `'Expected an ope_cllw_u64_65 (opf) value in json'`). Schema authors must keep the variant name aligned with what their Proxy actually emits for that column; the type system does not catch drift.

**Migration recipe.**

1. **Identify columns of type `public.encrypted_int4`**:

```sql
SELECT n.nspname || '.' || c.relname AS table_name, a.attname AS column_name
FROM pg_attribute a
JOIN pg_class c ON c.oid = a.attrelid
JOIN pg_namespace n ON n.oid = c.relnamespace
WHERE a.atttypid = 'public.encrypted_int4'::regtype;
```

2. **For each, choose a variant** based on what the Proxy emits for that column:
   - Storage only → `eql_v2_int4_ct`
   - Equality only (HMAC index term) → `eql_v2_int4_eq`
   - Equality + range with ORE blocks (`ob`) → `eql_v2_int4_ord_ore` or the default `eql_v2_int4`
   - Equality + range with OPE bytes (`opf`) → `eql_v2_int4_ord_ope`

3. **Run the type swap**:

```sql
ALTER TABLE t ALTER COLUMN c TYPE eql_v2_int4 USING c::jsonb::eql_v2_int4;
-- or _eq, _ord_ope, etc., per variant choice
```

4. **Install EQL 2.4.** The `DROP DOMAIN public.encrypted_int4` in `src/encrypted_domain/types.sql` now succeeds because no column depends on it.

**Range-index limitation (`_ord_ore`).** `_ord_ore` and the default `eql_v2_int4` use `eql_v2.compare_ore_block_u64_8_256`, which is PL/pgSQL and does not inline. Range queries (`<`, `<=`, `>`, `>=`) are seq-scan even with a functional btree on `eql_v2.ore_block_u64_8_256(col::jsonb)`. Equality remains btree-engaged via the hmac functional index.

Empirical EXPLAIN against an installed EQL 2.4 (single-row probe; seq-scan is the planner's choice independent of row count):

```
 Seq Scan on ore_probe  (cost=0.00..1.26 rows=1 width=204)
   Filter: (eql_v2.compare_ore_block_u64_8_256(value, value) < 0)
```

If range performance matters on large tables, choose `eql_v2_int4_ord_ope` instead — its range wrappers reduce to inlineable bytea lex-compare and engage a functional btree on the extractor expression.

**Domain ordering footgun.** `ORDER BY col` on a jsonb-backed domain follows native jsonb comparison, not the operator class. Sort by the extractor expression:

```sql
-- _ord_ore / default: sort by ORE bytes
ORDER BY eql_v2.ore_block_u64_8_256(col::jsonb)
-- _ord_ope: sort by OPE bytes
ORDER BY eql_v2.eql_v2_int4_ord_ope_ope_key(col::jsonb)
```

## Verification checklist

- [ ] `\dT public.eql_v2_int4*` shows the five new domains.
- [ ] `\dT public.encrypted_int4` returns "Did not find any relation named …" — the old type is gone.
- [ ] Every previously-int4 column has been altered to one of the new variants.
- [ ] Sample equality query on each variant shows `Index Scan` on the hmac functional btree.
- [ ] Sample range query on `_ord_ope` shows `Index Scan` on the OPE-key functional btree.
- [ ] Sample range query on `_ord_ore` shows `Seq Scan` (acknowledged limitation).

## Rollback

`git revert` the 2.4 release and reinstall the prior EQL release. User columns must be `ALTER TABLE ... ALTER COLUMN ... TYPE encrypted_int4 USING c::jsonb::encrypted_int4` reverted by hand — there is no automated downgrade.
